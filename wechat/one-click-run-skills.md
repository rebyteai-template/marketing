# Skill 需要一个浏览器

在万维网出现之前，信息是孤立的。

你的文件在你的电脑上。公司的数据在公司的服务器上。大学的论文在大学的局域网里。信息不是不存在，而是被锁在一台一台独立的机器里，彼此之间没有连接。

然后 Tim Berners-Lee 做了一件事：他定义了超链接协议。

超链接的本质极其简单——它就是一个地址。你点击一个链接，浏览器帮你找到那个地址对应的内容，然后呈现出来。就这样。没有什么魔法。但这个简单的机制，把全世界所有的信息连成了一张网。

不过，光有链接还不够。你还需要一个东西来"跑"这些链接。

这个东西叫浏览器。

超链接定义了"内容在哪里"，浏览器负责"把内容跑起来"。没有浏览器，超链接就只是一串字符。没有超链接，浏览器就无处可去。两者缺一不可。

这个结构——一个开放的寻址协议，加上一个通用的执行层——是过去三十年互联网最重要的架构。

我们认为，AI skill 正在重演同样的故事。

---

## Skill 就是链接

先说清楚一件事：一个 skill 到底是什么？

把所有包装拆掉，一个 skill 就是一段指令。它告诉 AI agent："这里有一组能力，代码在这个位置，请把它安装好，然后按照这个方式去用它。"

换句话说，skill 的核心就是一个地址加一份说明书。

这和超链接的结构惊人地相似。一个 URL 也不过是一个地址加上一个协议。URL 本身不包含任何内容——内容在服务器上。URL 做的事情只是告诉浏览器："去这个地方，把东西拿回来。"

Skill 也是一样。Skill 本身不"做"任何事。它只是描述了一种能力——代码可能托管在 GitHub 上，可能在某个 registry 里，可能在某个人的个人网站上。Skill 做的事情只是告诉 agent："去这个地方，把这个能力装上，然后用它。"

现在 AI 社区里已经有成千上万个 skill。深度研究、数据分析、代码生成、浏览器自动化、文档处理、股票分析——什么都有。而且这个数量还在爆炸式增长，因为写一个 skill 的门槛很低。任何一个开发者，花几个小时就能把自己的专业知识封装成一个 skill，然后发布到网上。

**信息已经在那里了。链接已经存在了。**

但大多数人用不了。

## 缺一个浏览器

问题不在 skill 本身。问题在于执行 skill 需要的东西太多了。

你需要一台配好环境的电脑。你需要安装正确版本的运行时。你需要配置 API key、环境变量。你需要一个 AI agent——Claude Code、Cursor、Codex，或者别的什么——然后告诉它去哪里找 skill，怎么装，怎么跑。

这就好像在 1990 年，你想看一份存在另一台服务器上的文档，你需要先搞清楚那台服务器的 IP 地址，然后手动建立 TCP 连接，然后解析 HTML，然后自己渲染出来。

技术上可行。实际上没有普通人会这么做。

所以才需要浏览器。浏览器把所有这些底层细节抽象掉了。你不需要知道 TCP 是什么，不需要知道 HTML 怎么解析，不需要知道服务器在哪个机房。你只需要点一下链接。

**Skill 现在就处于"没有浏览器"的阶段。**

能力在那里，数量在爆发，但只有懂技术的人能用。普通用户看到一个很酷的 skill，点进去一看——要装 Claude Code，要跑命令行，要配环境。然后就关掉了。

这不是用户的问题。这是基础设施的问题。

## 我们在做的事

Rebyte 做的事情，本质上就是给 skill 造一个浏览器。

当你说"我要用这个 skill"的时候，背后发生的事情是：

**第一，启动一台电脑。** 不是一个容器，不是一个 serverless function，是一台真正的云端虚拟机。有文件系统，有内存，有网络。这是 skill 运行的物理基础——就像浏览器需要一个操作系统来跑一样。

**第二，从网上动态拉取 skill。** Skill 的代码在 GitHub 上，在 skill 市场里，在任何一个公开的地址上。电脑启动后，自动定位、下载、安装。你不需要提前"部署"，就像你不需要提前把网页下载到本地才能看。

**第三，用 agent 执行。** 一个 AI agent 读取 skill 的指令，调用它的能力，完成你的任务。整个过程你能实时看到 agent 在做什么。

三步。启动电脑，下载 skill，执行。

这和浏览器处理一个 URL 的过程是完全对称的：建立连接，下载内容，渲染页面。

关键在于：**用户不需要理解这三步中的任何一步。** 就像你点一个链接不需要理解 DNS 解析一样，你跑一个 skill 不需要理解虚拟机怎么启动。

## 开放协议，不是封闭平台

这里有一个很重要的设计选择。

我们没有要求所有 skill 都必须发布到 Rebyte 的平台上。Skill 可以在任何地方——GitHub、GitLab、个人网站、企业内网。Rebyte 做的事情只是"去那个地方把它拿回来跑"。

这和浏览器的哲学是一样的。浏览器不要求所有网页都托管在某一个平台上。它只是去你给的地址，把内容拿回来。内容在哪里是你的自由。

这意味着 skill 的生态是开放的。任何人都可以写 skill，发布在任何地方，然后任何人都可以通过 Rebyte 来运行它。skill 的作者不需要跟我们有任何关系。用 skill 的人也不需要知道 skill 代码托管在哪里。

这就是一个开放协议应该有的样子：**创作者和消费者之间不需要直接连接，协议本身就是桥梁。**

## 超链接的类比走到底

如果 skill 就是链接，执行环境就是浏览器，那"超链接"对应的是什么？

我们做了一个叫 **"Run on Rebyte"** 的嵌入式按钮。任何网站、博客、文档页面，加一行代码就能出现这个按钮：

```html
<script src="https://app.rebyte.ai/embed.js"
  data-prompt="用 deep-research skill 分析量子计算最新进展">
</script>
```

用户点一下，Rebyte 自动启动电脑、下载 skill、执行任务。

**这就是 skill 世界的超链接。**

一个 `<a href>` 标签把你带到一个网页。一个 `<script>` 标签把你带到一个正在运行的 skill。两者的用户体验完全一致：点一下，看到结果。

而且，就像超链接可以出现在任何网页上一样，这个按钮也可以出现在任何地方。Skill 作者可以把它放在自己的博客里，企业可以把它嵌入内部文档，教育平台可以把它集成到课程页面。每一个按钮都是一个入口，通向一个正在云端运行的 AI agent。

配置页面在 [rebyte.ai/embed](https://rebyte.ai/embed)，可以自定义按钮样式、追踪来源渠道。

## 为什么这件事现在发生

互联网用了十年才从"学术网络"变成"所有人的网络"。关键转折点不是网页变得更好了，而是浏览器变得足够简单了。Mosaic 出来之后，普通人第一次能"点一下"就上网。

AI skill 现在正处在类似的时刻。能力已经爆发了。开源社区每天都在产出新的 skill。但使用门槛还停留在"你得是个开发者"的阶段。

我们相信，接下来真正重要的不是让 AI 更聪明（虽然那也在发生），而是让 AI 的能力 **可达**。让每一个人，不管技术背景如何，都能用上这些能力。

这需要一个执行层。一个浏览器。一个点一下就能跑的东西。

我们在造这个东西。
